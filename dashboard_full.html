🔍 Searching

SEARCHING ALGORITHMS
Finding elements in data
Searching is the process of locating a specific element in a collection. The two fundamental approaches are Linear Search (brute-force, works on any array) and Binary Search (divide-and-conquer, requires sorted data).
🔍
Linear Search
Scans every element from left to right until the target is found or the array is exhausted. No pre-sorting required. Best for small or unsorted datasets.

Complexity:
Best O(1)Average O(n)Worst O(n)Space O(1)
Algorithm Pseudocode
PSEUDOCODE function linearSearch(arr, target): for i = 0 to arr.length - 1: if arr[i] == target: return i // found at index i return -1 // not found

Binary Search
Divide array in half each time. Compare mid-element with target — if smaller go left, if larger go right. Eliminates half the search space each step. Requires sorted array.

Best O(1)Average O(log n)Worst O(log n)Space O(1)
Pseudocode
PSEUDOCODE function binarySearch(arr, target): left = 0, right = arr.length - 1 while left <= right: mid = (left + right) / 2 if arr[mid] == target: return mid else if arr[mid] < target: left = mid + 1 else: right = mid - 1 return -1

Theory Q
Why does Binary Search need sorted data?
Because the algorithm decides to go left or right based on comparison with mid. If the array is unsorted, "less than mid" does NOT mean "element is in left half" — so the logic breaks completely.
Theory Q
When to use Linear vs Binary?
Use Linear when data is unsorted, very small, or stored in a linked list. Use Binary when data is sorted and stored in an array/structure that allows O(1) random access by index.

SORTING ALGORITHMS
Arranging data in order
Sorting algorithms reorganize data in ascending or descending order. They differ by time complexity, stability (preserve equal element order), and in-place operation (O(1) extra memory).
↕️
Bubble Sort O(n²)
Repeatedly swap adjacent elements if they are in the wrong order. Each full pass "bubbles" the largest unsorted element to its correct position at the end.
Stable ✓  |  In-place ✓  |  Best for: nearly sorted small arrays
PSEUDOCODE for i = 0 to n-1: for j = 0 to n-i-2: if arr[j] > arr[j+1]: swap(arr[j], arr[j+1]) // bubble up

Merge Sort O(n log n)
Divide & Conquer. Recursively split array in half until single elements. Then merge pairs in sorted order.
Stable ✓  |  NOT in-place O(n) space
PSEUDOCODE function mergeSort(arr): if len(arr) <= 1: return arr mid = len(arr) / 2 L = mergeSort(arr[0..mid]) R = mergeSort(arr[mid..end]) return merge(L, R) function merge(L, R): result = [] while L and R not empty: if L[0] <= R[0]: append L[0] else: append R[0] append remaining
Split trace: [38,27,43,3] → [38,27] [43,3] → [38][27] [43][3]
Merge trace: [27,38] [3,43] → [3,27,38,43] ✓
Heap Sort O(n log n)
Build a Max-Heap from the array. Repeatedly extract the maximum (root) and place it at the end, then re-heapify.
Stable ✗  |  In-place ✓  |  Guaranteed O(n log n)
PSEUDOCODE // Phase 1: Build Max-Heap for i = n/2-1 downto 0: heapify(arr, n, i) // Phase 2: Extract elements for i = n-1 downto 1: swap(arr[0], arr[i]) // put max at end heapify(arr, i, 0) // re-heapify function heapify(arr, n, i): largest = i l = 2*i+1, r = 2*i+2 if l < n and arr[l] > arr[largest]: largest=l if r < n and arr[r] > arr[largest]: largest=r if largest != i: swap; heapify(arr, n, largest)
Counting Sort O(n+k)
Non-comparison sort. Count occurrences of each value, then reconstruct array using cumulative counts. Works only with integers in a known range [0..k].
Stable ✓  |  Space O(n+k)
EXAMPLE: [4,2,2,8,3,3,1] Count array (size 9): idx: [0,1,2,3,4,5,6,7,8] cnt: [0,1,2,2,1,0,0,0,1] cumulative: [0,1,3,5,6,6,6,6,7] Output: [1,2,2,3,3,4,8] ✓
Radix Sort O(d·n)
Sort digit-by-digit from least significant to most significant using a stable sort (counting sort) at each digit position. d = number of digits.
Stable ✓  |  Space O(n+k)  |  Great for large integer arrays
EXAMPLE: [170,45,75,90,802,24,2,66] Sort by ones: [170,90,802,2,24,45,75,66] Sort by tens: [802,2,24,45,66,170,75,90] Sort by hundreds:[2,24,45,66,75,90,170,802] ✓

📚 Stack

LIFO DATA STRUCTURE
Last In, First Out
A Stack is a linear data structure that follows the LIFO principle — the last element inserted is the first to be removed. Like a stack of plates. Core operations: push (insert), pop (remove), peek/top (view top), isEmpty.
📚
Push — O(1)
Insert element on top. Increment top pointer.
Stack: [] → push(A) → [A] → push(B) → [A,B] → push(C) → [A,B,C]
function push(stack, x): stack.top++ stack[top] = x
Pop — O(1)
Remove and return the top element. Decrement top pointer. Error if stack is empty (underflow).
function pop(stack): if isEmpty(stack): raise "Underflow" val = stack[top] stack.top-- return val
Peek / Top — O(1)
View the top element without removing it. Essential for expression evaluation and parsing.
function peek(stack): if isEmpty(stack): raise "Empty" return stack[top]
Polish Notation — Three Forms
Infix (Normal)
(A + B) × C
Operator between operands. Requires parentheses and precedence rules.
Prefix (Polish)
× + A B C
Operator before operands. No parentheses needed. Evaluated right-to-left.
Postfix (Reverse Polish)
A B + C ×
Operator after operands. Directly evaluable with a single stack pass.

Theory Q
Why is postfix easier to evaluate than infix?
Infix requires knowing operator precedence (+, ×, ÷) and handling parentheses. Postfix is evaluated with a single left-to-right pass using a stack — no precedence rules needed at all.
Theory Q
Real-world uses of stacks?
Browser Back button, function call stack (recursion), undo/redo in text editors, syntax checking (matching brackets), compiler expression evaluation, DFS graph traversal.

🚶 Queue

FIFO DATA STRUCTURE
First In, First Out
A Queue inserts at the rear (enqueue) and removes from the front (dequeue) — like a real-world queue/line. Variants: Simple Queue, Circular Queue, Priority Queue, Double-Ended Queue (Deque).
🚶
Simple Queue Operations
PSEUDOCODE Enqueue(Q, x): // Insert at rear if Q.rear == MAX-1: "Overflow" Q.rear++; Q[rear] = x Dequeue(Q): // Remove from front if Q.front > Q.rear: "Underflow" val = Q[front]; Q.front++; return val
1
Enqueue A,B,C → front=0, rear=2
A←front
B

Dequeue → returns A. front=1
B←front

Circular Queue
Problem with simple queue: once rear=MAX-1 even if front slots are free, we can't enqueue (wasted space).
Circular Queue wraps around: rear = (rear+1) % MAX, reusing freed slots.
CIRCULAR Enqueue(Q, x): if (rear+1) % MAX == front: "Full" rear = (rear + 1) % MAX Q[rear] = x Dequeue(Q): if front == rear: "Empty" front = (front + 1) % MAX return Q[front]
Example (MAX=5): After enqueuing A,B,C,D and dequeuing A,B → slots 0,1 are free. Simple queue fails! Circular wraps: enqueue E at index 0, F at index 1. No wasted space.

Theory Q
Why does a simple queue waste space?
Once the front pointer moves forward, those memory slots at the beginning can never be reused even though they are empty, because the rear pointer can only move forward. Circular Queue solves this by using modulo arithmetic.
Theory Q
Real-world uses of queues?
CPU process scheduling, print spooler, BFS graph traversal, network packet buffering, keyboard input buffer, ticket booking systems, call center phone queues.

🔗 Linked Lists

DYNAMIC LINEAR STRUCTURE
Nodes connected by pointers
A linked list is a collection of nodes where each node stores data and a pointer to the next node. Unlike arrays, linked lists don't require contiguous memory — insertion/deletion is O(1) if you have the pointer.
🔗
Singly Linked List
Each node has data + one next pointer. Traversal only in one direction. Last node points to NULL.

[10|→] → [20|→] → [30|→] → [40|NULL]
HEAD↑
Doubly Linked List
Each node has data + prev + next pointers. Bidirectional traversal. Allows O(1) deletion if node pointer is known.

NULL←[10]→←[20]→←[30]→NULL
HEAD↑                TAIL↑
Circular Linked List
Last node's next points back to the head. Useful for round-robin scheduling and continuous loops.

[10|→] → [20|→] → [30|→]
 ↑←←←←←←←←←←←←←←←←←←↙
Insertion — 3 Cases
①
Insert at head: Create new node, set new.next = head, update head = new. O(1).
[NEW]→[A]→[B]→[C]→NULL
②
Insert at tail: Traverse to last node, set last.next = new. O(n) for singly, O(1) if tail pointer stored.
[A]→[B]→[C]→[NEW]→NULL
③
Insert at position k: Traverse to node k-1, set new.next = node[k], node[k-1].next = new. O(k).
[A]→[B]→[NEW]→[C]→NULL
Deletion — 3 Cases
①
Delete head: head = head.next. Free old head. O(1).
~~[A]~~→[B]→[C]→NULL   head=[B]
②
Delete tail: Traverse to node n-2, set node[n-2].next = NULL. O(n).
[A]→[B]→NULL   ~~[C]~~
③
Delete node with value X: Find node before X, set prev.next = X.next. Free X. O(n).
[A]→~~[B]~~→[C]: set A.next = C
Search in Linked List — O(n)
Cannot use binary search (no random access by index). Must traverse from head node by node.
PSEUDOCODE function search(head, target): current = head position = 0 while current != NULL: if current.data == target: return position // found! current = current.next position++ return -1 // not found
ADVANTAGES ✓
Dynamic size — grows/shrinks at runtime
O(1) insertion/deletion at head
No memory wastage (allocates as needed)
Efficient for deque, stack, queue implementation
DISADVANTAGES ✗
No random access — O(n) to reach index k
Extra memory for pointers
Cache unfriendly (non-contiguous memory)
Reverse traversal requires doubly-linked

# Hash Functions

HASH TABLE
O(1) key-to-value mapping
A hash function maps a key to an index in an array (hash table). Goal: distribute keys uniformly. Collision = two keys map to same index. Two strategies: Open Hashing (Chaining) and Closed Hashing (Open Addressing).
#
Hash Function Design
A good hash function should:
→ Be fast to compute — O(1)
→ Distribute keys uniformly across all slots
→ Minimise collisions
→ Deterministic — same key always → same index
DIVISION METHOD h(key) = key % table_size // Example: table_size = 7 h(25) = 25 % 7 = 4 h(36) = 36 % 7 = 1 h(10) = 10 % 7 = 3 h(17) = 17 % 7 = 3 ← collision!
Open Hashing (Chaining)
Each slot holds a linked list. Colliding keys are chained in the same slot's list. Table never gets "full".
CHAINING — table[7], keys: 25,36,10,17,43 slot 0: [empty] slot 1: [36] → NULL slot 2: [43] → NULL slot 3: [10] → [17] → NULL ← chain! slot 4: [25] → NULL slot 5: [empty] slot 6: [empty] Search(17): h(17)=3 → traverse chain [10→17] ✓
Closed Hashing (Open Addressing) — 3 Probing Methods
LINEAR PROBING
If h(k) is full, try h(k)+1, h(k)+2, ... (mod n). Simple but causes primary clustering.
h(k,i) = (h(k) + i) % n
QUADRATIC PROBING
Probe at h(k)+1², h(k)+2², h(k)+3². Reduces clustering but may not probe all slots.
h(k,i) = (h(k) + i²) % n
DOUBLE HASHING
Use a second hash function for step size. Best distribution, eliminates clustering.
h(k,i) = (h1(k) + i·h2(k)) % n
Theory Q
Open vs Closed Hashing — when to choose?
Open (Chaining): better when load factor > 0.7, deletions are frequent, or table size is hard to predict.
Closed (Open Addressing): better cache performance (all data in one array), no pointer overhead, good when load factor < 0.5.
Theory Q
What is Load Factor?
Load Factor α = n/m, where n = number of keys stored, m = number of slots. When α is too high (e.g., >0.75), collisions increase and performance degrades. Most hash tables rehash (double size) when α exceeds a threshold.

🌳 Trees

HIERARCHICAL STRUCTURE
Nodes, edges, and balance
A tree is a non-linear data structure of nodes connected by edges with no cycles. Variants differ by balance guarantees: BST (no guarantee), AVL (height-balanced), Red-Black (color-balanced). All operations O(log n) in balanced trees.
🌳
Binary Search Tree (BST)
Property: Left subtree keys < root < Right subtree keys. Applies recursively to every subtree.

Search O(log n)Insert O(log n)Delete O(log n)
Worst case O(n) if unbalanced (sorted input)
INSERT function insert(node, val): if node == NULL: return newNode(val) if val < node.val: node.left = insert(node.left, val) else if val > node.val: node.right = insert(node.right, val) return node
TRAVERSALS on BST [50,30,70,20,40,60,80]
Inorder (L→Root→R) = 20 30 40 50 60 70 80 ← sorted!
Preorder (Root→L→R) = 50 30 20 40 70 60 80
Postorder (L→R→Root) = 20 40 30 60 80 70 50
BST STRUCTURE
AVL Tree — Self-Balancing BST
Balance Factor = height(left) − height(right). Must be -1, 0, or +1 for every node. If |BF| > 1, perform rotation to rebalance.

All ops guaranteed O(log n)
4 Rotation Cases:
LL (Right Rotate): inserted in left of left child
RR (Left Rotate): inserted in right of right child
LR (Left-Right): left rotate child, then right rotate root
RL (Right-Left): right rotate child, then left rotate root
RIGHT ROTATION (LL case) function rightRotate(y): x = y.left T2 = x.right x.right = y y.left = T2 // Update heights y.height = max(h(y.left), h(y.right)) + 1 x.height = max(h(x.left), h(x.right)) + 1 return x // x becomes new root
AVL INSERTION EXAMPLE: insert 10,20,30
Step 1: Insert 10 → [10] BF=0 ✓
Step 2: Insert 20 → [10(R:20)] BF(10)=-1 ✓
Step 3: Insert 30 → [10(R:20(R:30))] BF(10)=-2 ✗
→ RR case! Left rotate at 10
→ Result: [20(L:10,R:30)] BF=0 ✓

Red-Black Tree
Properties (5 rules):
1. Every node is RED or BLACK
2. Root is always BLACK
3. NULL leaves (NIL) are BLACK
4. If a node is RED, both children are BLACK
5. All paths from a node to NIL have the same number of BLACK nodes
Height ≤ 2·log₂(n+1)
Guarantees O(log n) for all operations.
Rebalancing uses rotations + recoloring.
Less strictly balanced than AVL → faster insertions.
Used in: Java TreeMap, C++ std::map, Linux scheduler.

🕸️ Graphs

GRAPH ALGORITHMS
Vertices, edges, and paths
A graph G=(V,E) consists of vertices (nodes) and edges (connections). Graphs model networks, maps, social connections, dependencies. Key representations: Adjacency Matrix O(V²) space, Adjacency List O(V+E) space.
🕸️
Adjacency Matrix
2D array where matrix[i][j]=1 if edge exists. O(1) lookup, O(V²) space. Good for dense graphs.
GRAPH: A-B, A-C, B-D, C-D A B C D A [ 0 1 1 0 ] B [ 1 0 0 1 ] C [ 1 0 0 1 ] D [ 0 1 1 0 ]
Adjacency List
Array of linked lists. Each node stores its neighbors. O(V+E) space. Best for sparse graphs.
SAME GRAPH A → [B, C] B → [A, D] C → [A, D] D → [B, C] // Space: O(V+E) vs O(V²) // For sparse graphs: huge savings!
BFS — Breadth-First Search
Explore level-by-level using a Queue. Visits all neighbors before going deeper. Finds shortest path in unweighted graphs.

Time O(V+E)Space O(V)
PSEUDOCODE function BFS(graph, start): visited = {} ; queue = [start] visited.add(start) while queue not empty: v = queue.dequeue() process(v) for neighbor in graph[v]: if neighbor not in visited: visited.add(neighbor) queue.enqueue(neighbor)
BFS TRACE — start from A
1
Queue: [A]. Visit A. Enqueue neighbors B,C.
Queue: [B, C]. Visited: {A}
2
Dequeue B. Visit B. Enqueue D (unvisited).
Queue: [C, D]. Visited: {A,B}
3
Dequeue C. Visit C. D already queued.
Queue: [D]. Visited: {A,B,C}
4
Dequeue D. Visit D. No new neighbors.
Queue: []. BFS Order: A→B→C→D
DFS — Depth-First Search
Explore as deep as possible using a Stack (or recursion). Backtracks when dead end reached. Used for cycle detection, topological sort, maze solving.

Time O(V+E)Space O(V)
RECURSIVE function DFS(graph, v, visited): visited.add(v) process(v) for neighbor in graph[v]: if neighbor not in visited: DFS(graph, neighbor, visited)
DFS TRACE — start from A
1
Visit A. Go to first neighbor B. Stack: [A]
2
Visit B. Go deep to D. Stack: [A,B]
3
Visit D. Check C (unvisited) → visit C. Stack: [A,B,D]
4
Visit C. All neighbors visited. Backtrack.
DFS Order: A→B→D→C
⭐ Dijkstra's Algorithm — Shortest Path (Weighted)
Finds the shortest path from a source vertex to all other vertices in a weighted graph with non-negative weights. Uses a priority queue (min-heap).
Time O((V+E) log V)Works with non-negative weights onlyFails with negative edges
PSEUDOCODE function Dijkstra(graph, src): dist[src]=0; dist[all others]=∞ pq = MinHeap(); pq.insert(src, 0) while pq not empty: u = pq.extractMin() for (v, weight) in graph[u]: if dist[u]+weight < dist[v]: dist[v] = dist[u] + weight pq.decreaseKey(v, dist[v]) return dist[]
STEP-BY-STEP TRACE — Source: A
Init
dist[A]=0, dist[B]=∞, dist[C]=∞, dist[D]=∞. PQ: [(A,0)]
1
Extract min: A(0). Check neighbors:
→ B: 0+4=4 < ∞ → dist[B]=4. PQ: [(B,4)]
→ C: 0+2=2 < ∞ → dist[C]=2. PQ: [(C,2),(B,4)]
2
Extract min: C(2). Check neighbors:
→ D: 2+1=3 < ∞ → dist[D]=3. PQ: [(D,3),(B,4)]
→ B: 2+5=7 > 4, no update.
3
Extract min: D(3). No shorter paths found. PQ: [(B,4)]
4
Extract min: B(4). Check D: 4+3=7 > 3. No update.
Final: A→A=0, A→B=4, A→C=2, A→D=3
Shortest path to D: A→C→D (cost 3) ✓
Bellman-Ford Algorithm — Handles Negative Weights
Relax all edges V-1 times. After V-1 relaxations, if any edge can still be relaxed → negative cycle detected.

Time O(VE)Space O(V)Handles negative weightsDetects negative cycles
PSEUDOCODE function BellmanFord(graph, src): dist[src]=0; dist[others]=∞ // Relax V-1 times for i = 1 to V-1: for each edge (u,v,w): if dist[u]+w < dist[v]: dist[v] = dist[u]+w // Check negative cycle for each edge (u,v,w): if dist[u]+w < dist[v]: return "Negative cycle!" return dist[]
Example — Graph with negative edge:
Edges: A→B(4), A→C(3), B→C(-2), C→D(1), B→D(5)

Pass 1: A→B=4, A→C=3
After relax B→C(-2): dist[C]=min(3, 4-2)=2
After relax C→D: dist[D]=min(∞, 2+1)=3

Pass 2: No improvements (converged)

Final: A=0, B=4, C=2, D=3 ✓
Dijkstra would give WRONG answer here because of negative edge B→C(-2)!
Why Dijkstra fails with negative edges
Dijkstra assumes: once a node is "finalized" (extracted from PQ), its distance is optimal. A negative edge can later reduce that distance, breaking this assumption.
Floyd-Warshall Algorithm — All-Pairs Shortest Paths
Finds shortest paths between every pair of vertices. Uses dynamic programming with intermediate vertices.
Time O(V³)Space O(V²)Handles negative weightsDetects negative cycles
PSEUDOCODE // dist[i][j] = initial edge weights // ∞ if no direct edge for k = 0 to V-1: // intermediate for i = 0 to V-1: for j = 0 to V-1: if dist[i][k] + dist[k][j] < dist[i][j]: dist[i][j] = dist[i][k] + dist[k][j] // Detect negative cycle: if dist[i][i] < 0: "Neg cycle!"
Example (4 vertices):

Theory Q
BFS vs DFS — when to use each?
BFS: when you need the shortest path (unweighted), or want to explore level by level (e.g., social network "degrees of separation").
DFS: when you need to explore all paths (backtracking, maze solving), detect cycles, or perform topological sort.
Theory Q
Why does Floyd-Warshall need O(V³)?
It uses 3 nested loops: for every intermediate vertex k, for every source i, for every destination j — check if going through k gives a shorter path. That's V × V × V = V³ operations.
Theory Q
How does Bellman-Ford detect negative cycles?
After V-1 relaxations, all shortest paths are finalized (a path has at most V-1 edges). Run one more pass: if any distance can still be reduced, it means there's a cycle that keeps getting shorter — a negative cycle.
Theory Q
What is relaxation in shortest path algorithms?
Relaxation is the process of checking: "Is the current known distance to vertex v worse than the distance through vertex u?" If dist[u] + weight(u,v) < dist[v], update dist[v]. This is the core operation in Dijkstra, Bellman-Ford, and Floyd-Warshall.
